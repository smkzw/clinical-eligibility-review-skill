# 临床试验入选/排除审核 Skill

这是一个面向临床试验医学监查的 Agent Skill，用于把研究方案、EDC listing、原始入排资料、OCR/VL 识别结果、人工审核台账整合成可追溯的入排审核流程。

它的目标不是让 AI 替代医学判断，而是帮助医学监查员更快、更系统地完成证据整理、逐条入排判断、风险提示、补充资料清单和审计台账。

## 这个 Skill 能做什么

这个 skill 会指导 Agent 完成以下工作：

1. 阅读研究方案，拆解入选标准和排除标准。
2. 如果研究方案同时包含多个阶段或队列，要求用户明确本次审核对应的阶段或队列。
3. 把每条入排标准转换成可审核规则，包括完整方案原文、所需证据、时间窗、阈值、单位、例外情况、复筛/复测逻辑和缺失资料提示。
4. 从 EDC listing 中提取与入排审核有关的数据。
5. 对受试者原始资料进行 OCR/VL 识别，包括文字 PDF、扫描 PDF、图片、照片、Word 派生文档等。
6. 为每个受试者建立证据包。
7. 使用 Agent 或经批准的大模型逐条审核。
8. 生成临床监查员可阅读的 HTML 报告。
9. 生成 Excel 形式的审核台账。
10. 支持补充资料后的二次审核和增量更新。

## 适合谁用

适合：

- 医学监查员；
- CRA/项目医学支持人员；
- 申办方临床运营团队；
- 质量管理和核查准备团队；
- 需要做入排源数据核查的临床数据/医学团队。

不需要使用者懂 AI 技术。使用者主要负责确认方案规则、判断证据是否合理、处理需要医学判断或溯源验证的问题。

## 重要边界

这个 skill 不能替代：

- 研究者医学判断；
- 医学监查最终签署或确认；
- 源数据核查；
- GCP/ICH 要求；
- 申办方 SOP；
- 法规或伦理层面的正式决策。

AI 输出应作为审核辅助材料，最终仍需人工确认。

## 常见输入

通常需要：

- 研究方案，Word 或 PDF；
- EDC data listing，通常为 Excel；
- 受试者原始资料，例如筛选病历、基线病历、检验报告、检查单、知情同意相关记录、既往病历、影像/肺功能报告等；
- 人工入排审核记录，如已有；
- 后续补充资料。

## 常见输出

建议输出：

- 结构化入排审核规则；
- 受试者证据包；
- 逐条规则审核结果；
- 受试者整体审核结果；
- HTML 审核报告；
- Excel 审核台账；
- 审核运行和补充资料审计记录。

## HTML 报告应该怎么看

HTML 报告是给医学监查员快速抓重点的，不是技术运行日志。

每个受试者页面建议只显示：

- 筛选号；
- 中心代号及名称；
- 年龄；
- 性别；
- ICF 签署日期；
- 整体审核结果；
- 重点提示；
- 逐条入排审核卡片。

HTML 报告不应显示模型名称、运行日志、内部证据编号、机器字段名、debug 信息或指令类文字。

## 证据分级

优先使用原始一手资料：

- 筛选/基线病历；
- 已签署或已记录的原始源文件；
- 检验报告；
- 检查单；
- 影像、肺功能等报告；
- 其他直接来源于临床现场或实验室的原始资料。

需要提示“需验证”的二手或加工资料：

- 仅 EDC listing；
- 仅邮件或沟通记录；
- 仅 EDC 加邮件/沟通记录，缺少一手原始资料；
- OCR 识别质量影响关键事实；
- 文件身份或受试者编号存在疑问。

如果某条规则只是基于这些二手资料通过，报告里应写成“通过（需验证）”，不能当作普通通过。

EDC 中“是否满足所有入选标准且不满足任一排除标准”这类总判断字段，不能作为任何单条入排标准的证据。如果某条规则只有这种总判断字段，而没有其他可定位原始资料或有效结构化证据，应按“证据不足”处理，而不是按“通过（需验证）”处理。

## 规则卡片应如何展示

每条规则建议：

- 折叠标题显示规则编号和简短中文医学标题；
- 展开后显示完整研究方案原文；
- 显示审核结果；
- 用简短自然语言解释为什么通过、为什么不通过、为什么证据不足或为什么需验证；
- 显示原始证据定位；
- 证据定位到文件名和页码，或 EDC 子表；
- 证据文字保留原文，不重新加工；
- 对真正对应本条规则的关键语句加粗或高亮。

高亮必须精细。遇到 quote 中有省略号、OCR 换行或 AI 归一化表述时，应回到原始资料中定位并高亮真实原文片段，不能因为精确匹配失败就把整段证据全部高亮。

候选证据不等于可展示证据。报告展示前必须再次确认这段原文是否真的证明了当前这条标准所要求的临床事实。比如：

- 评分达标标准要展示真实评分、分项、总分、日期或访视，不能把评分提醒、日记卡说明当成达标证据；
- 诊断和病史标准要展示诊断、病史年限和必要检查结果，不能只展示疾病相关的泛泛描述；
- 治疗反应标准要展示既往治疗、疗效不佳或矛盾核实信息，不能只展示无关病史；
- 有时间窗的排除标准要展示本条时间窗对应的完整原文，不能把相邻排除标准的句子一起标亮。

如果原始病历里找不到当前标准需要的字段，应主动检查对应 EDC 子表或把该条列为证据不足，而不是随意高亮不相关文字。

多组成要件的标准要逐项展示证据。比如过敏性鼻炎诊断/病史类标准，应同时展示诊断和病史年限原文，以及具体阳性的过敏原/sIgE 结果行；“6级重度过敏”这类浓度解释或分级表不是具体检测结果，不能代替过敏原结果行。

实验室或感染筛查排除标准应展示真实实验室/病毒学/感染筛查结果行，不能展示相邻的过敏原解释表。OCR 在句子中间插入单个换行时，报告不应因此截断病史年限、药物名称、生物制剂名称或手术条款。

EDC 证据展示前应去掉 listing 管理字段，例如项目编号、表单编号、受试者编号、姓名缩写、中心编号/名称、行号、最后修改时间等。页面上只保留非空临床字段，例如结果、药物名称、用药原因、日期、临床评估和关键是否值。

这样监查员打开一条规则时，可以同时看到“方案要求是什么”“判断依据是什么”“证据原文在哪里”。

## 左侧筛选和状态提示

推荐左侧栏包括：

- 研究中心下拉筛选；
- 筛选号下拉筛选；
- 审核状态多选筛选。

状态建议包括：

- 不通过；
- 证据不足；
- 证据冲突；
- 通过需验证；
- 通过。

视觉强调程度应按临床优先级排列：

1. 不通过；
2. 证据不足或证据冲突；
3. 通过需验证；
4. 通过。

“通过（需验证）”要能被看到，但不能比“证据不足”或“证据冲突”更醒目。

左侧受试者列表建议采用严肃的状态卡片样式：固定或 sticky 侧栏、浅色受试者卡片、左侧状态色条、克制的临床配色，以及项目品牌色作为小面积强调。右侧报告滚动时，左侧筛选栏应保持固定，方便监查员随时切换受试者。

## Excel 台账

Excel 台账用于记录 HTML 报告里不适合展示的运行和追踪信息。

建议至少包含：

- 审核汇总；
- 规则明细；
- 运行台账；
- 人工对照；
- 补充资料清单。

台账中可以保留模型路线、OCR 模型、隐私模式、运行触发原因、补充资料、二次审核等信息。

## 安装方式

在支持本地 skill 的 Codex 或 Agent 环境中，可以把 `SKILL.md` 放入 skill 目录。例如：

```bash
mkdir -p ~/.codex/skills/clinical-eligibility-review
cp SKILL.md ~/.codex/skills/clinical-eligibility-review/SKILL.md
```

之后在对话中要求 Agent 使用 `clinical-eligibility-review` skill 进行临床试验入排审核即可。

## 示例配置

见 `examples/project-config.example.yaml`。

项目配置应明确：

- 本次审核的阶段或队列；
- 隐私模式；
- OCR/VL 模型；
- 审核大模型或 Agent 审核路线；
- 是否允许没有原始资料的受试者仅基于 EDC 审核；
- 输出工作区；
- 报告语言。

安全默认值是：没有原始资料的受试者不进入正式审核报告，除非项目配置明确允许。

## 使用前 QC 清单

正式使用报告前，建议检查：

- 所有入选/排除标准是否完整列出；
- 阶段或队列是否明确；
- 受试者筛选号和原始资料文件夹是否对应；
- OCR 质量问题是否影响关键事实；
- 仅 EDC、仅邮件、仅 EDC 加邮件的通过项是否都标记为需验证；
- EDC 入排总判断字段是否未被当成单条标准证据；
- 不通过、证据不足、证据冲突、通过需验证是否默认展开；
- 普通通过项是否默认折叠；
- HTML 是否没有模型名、运行日志、内部证据 ID 和机器字段；
- 高亮证据是否真正对应当前标准，而不是只因为候选证据里出现了相关关键词；
- 多组成要件的标准是否展示了所有必要组成，而不是只展示其中一部分；
- 评分类标准是否引用真实评分记录，而不是引用评分提醒或日记卡说明；
- 实验室/感染筛查标准是否展示真实检验行，而不是展示过敏原解释表；
- 右侧报告滚动时左侧筛选栏是否保持固定；
- Excel 台账能否正常打开；
- 医学监查员是否已进行人工复核。

## 隐私和合规提醒

临床试验资料可能包含个人信息和申办方保密信息。除非项目配置、组织制度、研究合同和适用法规均允许，不应把资料发送到远程模型。

本地 OCR/VL 或本地大模型可以降低数据外传风险，但仍需要权限控制、操作记录和审计意识。
